AI开发者规范手册

一、通用原则

代码质量：始终编写干净、可维护且文档齐全的代码
安全第一：在所有实现中考虑安全影响
性能优化：在保持可读性的同时优化效率
测试覆盖：为关键功能编写单元测试
文档清晰：提供清晰的注释和文档

二、版本号规范

版本号格式
采用语义化版本控制 (Semantic Versioning) v2.0.0 格式：主版本号.次版本号.修订号

版本号含义
主版本号 (Major)：当做了不兼容的 API 修改时递增
次版本号 (Minor)：当增加了向后兼容的功能时递增
修订号 (Patch)：当修复了向后兼容的问题时递增

预发布版本
alpha：内部测试版本
beta：外部测试版本  
rc：发布候选版本 (Release Candidate)

版本号示例
v1.0.0          - 初始正式版本
v1.0.1          - 补丁更新
v1.1.0          - 新功能添加
v2.0.0          - 不兼容的重大更新
v1.0.0-alpha    - 预发布 alpha 版本
v1.0.0-beta.1   - 预发布 beta 版本
v1.0.0-rc.1     - 发布候选版本

版本发布时机
主版本：重大重构、API 不兼容变更、架构升级
次版本：新增功能、向后兼容的增强
修订版本：错误修复、性能优化、文档更新

三、Git 提交规范

提交时机
功能完成：完成逻辑功能或用户故事后
错误修复：修复特定问题或错误时
代码重构：完成代码重构后
集成节点：与外部服务集成前
里程碑节点：在约定的项目里程碑处
破坏性变更：在进行破坏性变更前立即提交
配置更新：更新配置文件时
依赖更新：更新依赖项或包后

提交频率
小而逻辑：目标是代表单一更改的原子提交
不过于频繁：避免对小拼写错误或格式更改进行提交
不过于稀少：不要积累大量未提交的工作
定期间隔：每小时专注工作至少提交一次

提交消息格式
(): 

类型值
feat：为用户添加新功能
fix：修复用户错误
docs：仅文档更改
style：格式化、缺少分号等（无代码更改）
refactor：重构生产代码
test：添加测试，重构测试
chore：维护任务、构建过程、辅助工具
perf：性能改进
build：构建系统或外部依赖更改
ci：CI 配置文件和脚本更改
revert：回滚先前的提交

范围示例
api、database、ui、auth、payment、config、deps

主题指南
使用祈使语气（"添加"而不是"添加了"或"已添加"）
以小写字母开头
结尾不加句号
最多 50 个字符
具体且描述性强

正文指南
解释什么和为什么，而不是怎么做
在 72 个字符处换行
如需要可使用项目符号
使用 #issue_number 引用相关问题

页脚指南
如适用，包含 BREAKING CHANGE 注释
引用关闭的问题
如需要包含作者信息

四、分支管理

分支命名

拉取请求流程

压缩提交：合并前清理提交历史
代码审查：合并前需要同行审查
CI 检查：确保所有测试通过
文档更新：更新相关文档

五、代码规范

命名规范

变量命名
驼峰命名法 (camelCase)：局部变量、函数参数
帕斯卡命名法 (PascalCase)：类型定义、类名、接口名
大写下划线 (UPPER_SNAKE_CASE)：常量、枚举值

文件命名
短横线命名法 (kebab-case)：文件名、目录名
首字母大写 (PascalCase)：组件文件

函数命名
动词开头：描述函数的行为
清晰明确：准确描述函数功能
一致性：同类型函数使用相同前缀

类命名
名词或名词短语：描述类的实体
帕斯卡命名法：首字母大写
单数形式：通常使用单数形式

代码结构

文件组织
相关功能分组：将相关功能放在适当的模块中
导入顺序：标准库 > 第三方 > 本地导入
函数长度：尽可能保持函数在 50 行以内
类大小：限制类最大 200-300 行

代码风格
缩进：统一使用 2 或 4 个空格（项目内保持一致）
行长度：建议不超过 80-120 字符
空行：合理使用空行分隔逻辑块
括号：保持一致的括号风格

错误处理

异常处理
总是处理错误：永不忽略错误返回
特定错误类型：适当时创建自定义错误类型
日志记录：带上下文记录错误，但避免敏感数据
用户反馈：向用户提供有意义的错误消息

验证和校验
输入验证：验证所有用户输入
边界检查：检查数组和字符串边界
类型检查：确保类型安全
状态验证：验证前置条件和后置条件

安全实践

输入安全
SQL 注入防护：使用参数化查询
XSS 防护：渲染前清理输出
CSRF 防护：实施 CSRF 令牌
文件上传：验证文件类型和大小

认证授权
身份验证：实施适当的身份验证
权限控制：实现细粒度的授权
会话管理：安全的会话处理
密码安全：使用强哈希算法

六、测试规范

测试覆盖率
最低 80%：整体代码覆盖率要求
关键路径：业务关键代码 100% 覆盖
边界情况：测试边界条件和错误状态

测试结构
安排-执行-断言 (AAA)：遵循 AAA 模式
描述性名称：使用清晰具体的测试名称
隔离性：每个测试应独立运行
快速执行：测试应快速运行

测试类型
单元测试：测试单个函数或方法
集成测试：测试组件间交互
端到端测试：测试完整用户流程
性能测试：测试系统性能指标

七、文档规范

代码注释
解释原因：不只是说明代码做什么
复杂逻辑：记录复杂算法和业务逻辑
待办事项：使用 TODO 格式并引用问题编号
API 文档：彻底记录公共接口

外部文档
README：包括设置说明和使用示例
API 文档：维护当前的 API 文档
架构文档：记录系统架构决策
部署指南：包括部署和配置指南

注释风格
行内注释：使用 // 进行行内注释
块注释：使用 /* */ 进行多行注释
文档注释：使用 /** */ 进行 API 文档
注释位置：注释在被注释代码上方

八、性能考虑

优化优先级
数据库查询：优化慢查询，避免 N+1 问题
缓存策略：实施适当的缓存策略
资源使用：监控内存和 CPU 使用情况
响应时间：在可能的情况下保持亚秒响应时间

性能监控
指标收集：跟踪关键性能指标
告警设置：为性能下降设置告警
性能分析：定期进行性能分析
基准测试：建立性能基准

九、协作规范

代码审查
建设性反馈：关注改进建议，而非批评
最佳实践：检查是否符合标准
安全性：审查潜在的安全问题
性能：考虑性能影响

沟通方式
清晰问题：写出详细的问题描述
状态更新：定期更新进度
阻塞报告：立即报告阻塞问题
知识分享：分享学习和发现

十、部署指南

部署前
测试完成：完成所有测试套件
文档更新：更新部署文档
备份准备：确保备份程序就绪
回滚计划：准备好回滚策略

部署后
健康监控：验证应用程序健康状况
用户通知：如有需要通知用户更改
性能检查：监控性能指标
问题跟踪：关注部署后的新增问题

十一、环境管理

环境区分
开发环境 (dev)：本地开发和功能测试
测试环境 (test)：集成测试和功能验证
预生产环境 (staging)：生产前最终验证
生产环境 (prod)：正式对外服务

配置管理
环境变量：使用环境变量管理配置
配置文件：不同环境使用不同配置文件
密钥管理：安全存储敏感信息
版本控制：配置文件纳入版本控制
